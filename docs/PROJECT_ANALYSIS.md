# Анализ проекта PT_LEP_unix

## 1. Текущее назначение

Проект реализует Ansible-роль для подготовки Linux-хостов к передаче событий безопасности в SIEM:
- контроль и конфигурация audit-подсистемы;
- доставка событий через syslog-стек;
- поддержка offline-инсталляции пакетов.

## 2. Сильные стороны

1. Практичный production-фокус: есть fallback на локальные пакеты.
2. Широкий покрываемый список ОС (Deb/RHEL/Ubuntu/Astra/Alt/REDOS).
3. Шаблоны маршрутизации событий на несколько SIEM-агентов.
4. Теги и разбиение task-файлов упрощают выборочный запуск.

## 3. Технические риски и ограничения

1. В `packages_dirs` есть расхождения с фактическими каталогами пакетов:
   - REDOS указывает `murom7`, в дереве пакетов есть `redos7`;
   - запись для `RED` не содержит `dist/ver/dir_name`.
2. Установка в `without_repos/*` использует `ignore_errors: True` без итоговой валидации, что может скрыть частичные сбои.
3. Определение syslog-демона по `pgrep` зависит от запущенного процесса; установленный, но не запущенный сервис может определиться неверно.
4. Сравнение версий местами строковое (`auditd_installed_ver.stdout < '3'`), что хрупко при нетипичном формате.
5. В репозитории есть inventory с реальными внутренними адресами, что повышает операционные риски при неконтролируемом доступе.

## 4. Рекомендуемые улучшения

### Приоритет P1
1. Исправить `packages_dirs` для REDOS/RED и синхронизировать с `files/packages`.
2. Добавить обязательную post-install проверку `auditctl`, `rsyslog -v`/`syslog-ng --version` в offline ветках.
3. Заменить определение syslog-демона с `pgrep` на проверку установленных сервисов и явный приоритет.

### Приоритет P2
1. Вынести повторяющуюся install-логику в reusable include/role helpers.
2. Перейти с raw shell-команд на idempotent модули там, где возможно.
3. Добавить `requirements.yml` для зависимостей (например `community.general`).

### Приоритет P3
1. Добавить CI: `ansible-lint`, syntax-check, smoke-run на test inventory.
2. Добавить Molecule-сценарии для ключевых дистрибутивов.
3. Отделить production inventory от репозитория роли.

## 5. Общий вывод

Кодовая база решает практическую задачу и пригодна к эксплуатации, но нуждается в небольшом цикле стабилизации вокруг:
- корректности offline mapping,
- детерминизма установки,
- автоматических проверок качества.
